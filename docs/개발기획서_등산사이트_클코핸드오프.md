# 개발 기획서 — 100대 명산 챌린지 도우미 (등산·둘레길 pSEO)
**클로드 코드(클코) 핸드오프 스펙 / v3 기준**

> ⚠️ **가장 먼저 읽을 것**: 이 프로젝트는 **validation-first**다. §3 Phase 0.5 검증 게이트를 통과하기 전에는 **수집기·스키마 본구현·페이지 빌드를 시작하지 말 것.** API 필드명과 데이터 품질은 실제 응답으로 확인되기 전까지 전부 "추정"이다. 검증 결과에 따라 컨셉이 축소·중단될 수 있다.

## 1. 프로젝트 컨텍스트

**무엇**: 산림청·관광공사 공공데이터로 한국의 산·등산로·둘레길 정보를 프로그래매틱 생성하는 Next.js 사이트.
**누구를 위해**: "100대 명산 완등"에 도전하는 등산객이 *집에서 "다음에 어느 산을 갈지" 계획하는 단계*를 돕는다. (등산 앱이 아니라 "선택·비교·계획" 웹사이트)
**왜 이 포지셔닝**: 트랭글 등 incumbent는 전부 GPS 트래킹 앱 → 산에서 켠다. "계획 단계"의 웹 검색 시장은 비어 있다.
**도메인**: **신규 독립 도메인** (bojo24와 분리. 주제 무관 → Site Reputation Abuse 회피 위해 서브도메인/디렉토리 금지).
**수익**: 애드센스 중심. 단기 폭발형 아님 — 연중 안정 + 재방문(완등 트래커) 자산.

### 절대 가드레일 (위반 시 프로젝트 리스크)
1. **상표**: "블랙야크/BAC" 명칭·로고 일절 사용 금지. **산림청 공식 100대 명산 목록**을 기준으로 하고, "100대 명산 챌린지"는 일반명으로만 표현. 특정 민간 프로그램 연계·사칭 암시 금지.
2. **안전 준-YMYL**: 산행은 안전 인접 주제. "쉽다/안전하다" 단정 금지. 모든 코스 페이지에 **안전 고지 블록 내장**(날씨·체력·장비·공식 통제정보 확인). 이걸 E-E-A-T 신호로 전환.
3. **Scaled Content Abuse 방어**: 각 페이지 **고유 데이터 포인트 3개+** 필수. "산 이름 빼도 콘텐츠가 유용한가? 아니오 → 발행 금지."
4. **데이터 2등급 현실 인정**: 전 코스 균일 정밀 난이도는 불가. GPX 보유 여부로 페이지 품질 등급을 나눈다(§5).

## 2. 기술 스택 (확정)

- Next.js 14 (App Router) + TypeScript **strict mode**
- Turso (Edge SQLite) + Drizzle ORM (raw SQL 최소화)
- Google Gemini API (Flash 기본) — 콘텐츠 보강
- Vercel (호스팅 + Cron 경량 작업) / GitHub Actions (대량 작업)
- 환경변수로 모든 시크릿 (.env)
- 디자인: 별도 산출된 `클로드디자인_프롬프트_등산사이트.md`의 토큰 시스템을 따른다(딥 포레스트 그린 팔레트, 등고선 모티프, 모바일 퍼스트).

| 한국관광공사 두루누비 | 15101974 | GPX 코스 수(≈284), 재배포 라이선스, 필드 |

### 검증 태스크 (TypeScript 스크립트, 순차)
```
T1. API 키 발급 확인 + 각 소스 1콜 raw 응답 덤프 → FIELD_MAP 작성
    (실제 응답 필드명을 기록. 추정 금지)
T2. 전국등산로표준데이터 N건(예 500) 샘플 →
    난이도/거리/소요시간 결측률·표기종류 집계
T3. 두루누비 코스 목록 조회 → GPX 보유 코스 수, 고도 추출 가능성 테스트
T4. 산림청 100대 명산 공식 목록 ↔ 표준데이터 산명 매핑률 측정
    (산명 표기 불일치·미스매치율)
T5. 라이선스 확인: 각 데이터셋 공공누리 유형, GPX/좌표 재배포 가부 → 보고
T6. 종합 의사결정 리포트 생성(아래 킬 기준 자동 판정)
```

### 킬 기준 (하나라도 해당 시 중단/재설계)
- **난이도·거리·소요시간 핵심 3필드 채움률 < 60%** 이고, 두루누비 GPX·공간데이터로도 보강 불가
- **100대 명산 ↔ 표준데이터 매핑률 < 70%** (척추가 무너짐)
- 두루누비/표준데이터 **재배포가 라이선스상 불가** (공공누리 4유형 등 변경금지·상업금지 조항)
- GPX 보유 코스가 너무 적어 "정밀 등급 페이지"가 사실상 코리아둘레길 284개뿐 → 단독 도메인 재고

**산출물**: `/scripts/phase05/` 아래 T1~T6 + `phase05-report.md`(결측률·매핑률·라이선스·GO/NO-GO).

## 5. DB 스키마 핵심 (Drizzle)

등산 도메인 맞춤. 주요 테이블:

```
data_sources   API 소스 관리(소스ID, 갱신주기, 라이선스유형)
mountains      산 마스터 (산명, 지역, 고도, is_top100, 좌표, 100대명산 메타)
courses        코스/등산로 (mountain_id FK, 거리, 소요시간, 난이도_raw,
               난이도_calc, gpx_available, tier(A정밀/B기본), 들머리, 교통)
trails         둘레길/숲길 (별도, 사계절 수요용)
pages          프로그래매틱 페이지
               (entity_type, status, quality_score, gate_passed,
                active_sections[], unique_points[], index_flag)
fillers        L2 비교·모음 (토픽클러스터 허브, index 우선, 수익 핵심)
evergreens     L3 가이드·안전 (권위)
hubs           카테고리/지역/100대명산 허브
personas       페르소나 (traveler + hiker)
publish_log    드립피드 추적
collect_checkpoints  수집 중단/재개
```

**핵심 필드 규칙**:
- `난이도_calc`: GPX 있을 때만 고도차·거리 가중 계산. 없으면 null, `난이도_raw`(지자체 표기) 사용 + "지자체 기준" 명시.
- `tier`: A(GPX정밀, index 우선) / B(표준 기본, 다수 noindex → 트래픽 발생 시 전환).
- `index_flag`: 계층형 인덱싱. 허브 + 비교/모음 + 상위 코스만 index, 단일 코스 하위는 noindex.

## 7. 품질 게이트 (발행 전 필수 통과)

draft → enriched → **quality_passed** → published (→ demoted: 감사 실패)

1. 고유 데이터 포인트 3개+ (100대명산 맥락 / 같은 산 코스 순위 / 교통 접근 등)
2. AI 코멘터리 200자+ (페르소나 동적 프롬프트, 특이점 기반)
3. 기존 글과 제목 유사도 < 30%
4. FAQ 3개+ (각 답변 50자+, AI 인용 최적화 위해 짧고 직접적)
5. 조건부 섹션 1개+ 활성화 (계절·교통·조망 등)
6. 비교 데이터 포함 (같은 산 코스 비교 또는 지역 평균 대비)
7. **안전 고지 블록 존재** (등산 도메인 필수 추가 기준)

게이트 실패 → rejected. 통과분만 드립피드.

## 8. SEO + GEO

- 온페이지: 동적 title·meta(Gemini), JSON-LD(Article·BreadcrumbList·FAQPage; LocalBusiness는 산엔 부적합 → **TouristAttraction/Place** 검토), 캐노니컬, OG/Twitter
- 테크니컬: 분할 sitemap(index분만), robots.txt(AI봇 허용: GPTBot·PerplexityBot·Claude-Web), ISR(허브·상위 사전빌드/나머지 on-demand), CWV(Suspense·스트리밍·이미지 최적화), 모바일 퍼스트
- GEO: `/llms.txt`(사이트 핵심 안내), SSR 필수(AI 크롤러 JS 미실행), FAQ 짧고 직접적
- 내부링크: 같은 지역 + 같은 100대명산 그룹 + 비교/모음/가이드 연결. **단일 코스(L1) → 비교·모음(L2)로 깔때기.**

## 9. 페르소나 (동적 프롬프트)

- `traveler`(정여행): 감성·생생 — 풍경·코스 분위기
- `hiker`(신규, 산행 도우미): 실용·신뢰 — 난이도·준비·안전. 단정·과장 금지.
- raw_data에서 특이점 자동 추출(고도 높음/계곡/조망/접근 편리 등) → 페르소나+특이점 기반 매번 다른 코멘터리. **의료성·치유 효과 표현 금지.**

## 10. 자동화

- Vercel Cron: `publish`(매일 06:00 드립피드), `collect`(증분)
- GitHub Actions: `bulk-collect`(체크포인트), `bulk-enrich`, `quality-gate`, `generate-filler`(비교/모음·계절 콘텐츠), `refresh-data`, `quality-audit`(발행 후), `generate-stats`
- 드립피드: 시드 10~20/일 → 성장 → 확장. 계층형 인덱싱(허브+상위20% index, 하위 noindex, 트래픽 시 전환)
